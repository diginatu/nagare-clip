# LLM Stage Report — Design

**Date:** 2026-06-14
**Status:** Approved (pending spec review)

## Problem

The pipeline runs many LLM stages (`text_filter` including its summary-LLM and
batch-halving retries, `summary`, `plan`, `director`, `guided_edit`). Each stage
retries a bounded number of times and degrades gracefully on failure. Today the
only record of *what happened* — how many retries a call took, what the LLM
actually returned, and what the prompt was at the time — is a scattering of
`logger.warning` / `logger.info` lines interleaved into a single run log. After a
run this is effectively impossible to review: you cannot tell which stage/unit
struggled, see the offending prompt/response, or count retries without grepping a
giant log.

## Goal

Produce a structured, human-reviewable **LLM report** under `output/llm_report/`
that, for every LLM call across every stage, records:

- the exact prompt (system + user messages) sent,
- the full raw response (or the error),
- how many attempts/retries the call took,
- the final outcome and the failure/drop reason,
- supporting metadata (stage, unit, model, temperature per attempt, timing).

This report becomes the audit trail; the existing `logger.warning` calls stay as
console breadcrumbs but are no longer the only record.

## Non-Goals

- Not changing retry behavior, temperature nudging, or any stage's output.
- Not a live dashboard or streaming view — files are written as stages run.
- Not capturing non-LLM stages (transcription, audio_silence, intervals,
  blender).
- No separate JSONL/event-stream file: the per-call markdown front-matter *is*
  the structured backing for the index.

## Output Layout

On by default. Root: `output/llm_report/` (configurable).

```
output/llm_report/
  index.md                   # regenerated table across all stages
  text_filter/
    <unit>.md                # unit e.g. "lines 11-20 (size 10)", "summary_llm"
  summary/
    <stem>.md                # one per segment_video call
    overall.md               # the reduce / generate_project_summary call
  plan/
    plan.md
  director/
    <stem>.md
  guided_edit/
    <stem>.md                # one section per director op applied
```

### Detail file format

Each detail `.md` is a markdown file with **YAML front-matter** above the body:

```markdown
---
stage: director
unit: my_video
model: qwen3.5:30b
attempts: 2
outcome: ok
reason: ""
started_at: 2026-06-14T10:31:02
duration_ms: 8421
---

# director — my_video

## Attempt 1 — temperature 0.10 — unparseable

### System prompt
...full system prompt...

### User prompt
...full user prompt (numbered transcript)...

### Response
...full raw LLM response...

### Result
unparseable: response has no 'ops' array

## Attempt 2 — temperature 0.30 — ok
...
```

Front-matter is the **structured source of truth** for the index. The body holds
the full prompt/response bodies for every attempt.

### `index.md`

Regenerated by scanning every detail file's front-matter. One row per detail
file:

| Stage | Unit | Attempts | Outcome | Reason | Detail |
|-------|------|----------|---------|--------|--------|
| director | my_video | 2 | ok | | [link](director/my_video.md) |
| guided_edit | my_video | — | dropped-items | 1 op unapplied: verify-fail | [link](guided_edit/my_video.md) |

A short header summarizes totals (e.g. counts of each outcome) so problem stages
are visible at a glance.

## Outcome vocabulary

A single closed set, recorded in front-matter (`outcome`) and per attempt:

- `ok` — succeeded (possibly after retries).
- `ok-empty` — a valid *empty* result accepted without retry (e.g.
  `{"ops": []}`, empty parts list, no directions).
- `llm-error` — connection/timeout/HTTP error from `_call_llm`.
- `unparseable` — hard parse failure (invalid JSON / missing required array).
- `verify-fail` — guided_edit `reconcile.verify_op` rejected the candidate.
- `dropped-items` — the call parsed but individual items were discarded (bad op
  lines, out-of-range plan index, malformed part, missing returned line, etc.).
  This captures the per-item warnings that are silent in the log today. The
  `reason` field names what was dropped and how many.

`outcome` on the *unit* (front-matter) is the final disposition; each *attempt*
also carries its own per-attempt outcome.

## Mechanism

### New module: `src/nagare_clip/llm_report.py`

A `Recorder` class plus a module-level index rebuilder.

```python
class Recorder:
    def __init__(self, stage: str, report_dir: Path | None, enabled: bool): ...
    def clear(self) -> None:
        """Wipe report_dir/<stage>/ at stage start (refresh-own-section rerun
        behavior). No-op when disabled."""
    def attempt(self, *, unit: str, attempt: int, total: int,
                messages: list[dict], response: str | None, error: str | None,
                outcome: str, reason: str, cfg: dict) -> None:
        """Buffer one attempt for `unit`."""
    def flush_unit(self, unit: str, *, final_outcome: str, final_reason: str,
                   model: str, started_at: str, duration_ms: int) -> None:
        """Write report_dir/<stage>/<unit>.md (front-matter + all attempts)."""

def rebuild_index(report_dir: Path) -> None:
    """Scan every <stage>/*.md front-matter, rewrite index.md."""

NULL_RECORDER  # a shared disabled Recorder; default for all stage functions
```

- A **no-op default** (`NULL_RECORDER`, `enabled=False`) keeps every stage
  function testable and keeps disabled runs zero-footprint.
- `unit` is slugified for the filename; the human-readable form is kept in
  front-matter.
- `flush_unit` aggregates the attempt buffer for a unit into a single file. The
  unit's `attempts` count and `outcome`/`reason` come from the last attempt (or
  are passed explicitly by the caller for cases like `guided_edit` where one
  file holds several ops — see below).

### Instrumentation sites

Each stage's retry loop takes an optional `recorder: Recorder = NULL_RECORDER`
and a `unit` label, and calls `recorder.attempt(...)` after each attempt, then
`flush_unit(...)` once. The sites (where the rich outcome is already known):

1. **`director_llm.generate_director_ops`** — unit = stem. Records per attempt:
   `llm-error` (exception), `unparseable` (`try_parse_...` returned None), or
   `ok`/`ok-empty`. If the parsed op list dropped items vs. the raw `ops` array,
   final outcome `dropped-items` with a count.
2. **`plan_llm.generate_plan`** — unit = `"plan"`. Same pattern; dropped indices
   → `dropped-items`.
3. **`summarize.segment_video`** — unit = stem; and
   **`generate_project_summary`** — unit = `"overall"`. Same pattern.
4. **`guided_edit/apply.apply_ops`** — unit = stem, **one file per stem** with a
   section per op. Each op's attempts are recorded; `verify_op` failures →
   `verify-fail`, exceptions → `llm-error`. Final unit outcome is `ok` if all
   ops applied, else `dropped-items` ("N op(s) unapplied"). `apply_ops` is the
   natural owner since it already loops ops × attempts.
5. **`llm_filter`** — the batch path uses recursive `_process_batch` with batch
   halving (not `llm_retry`). Unit = `"lines {a}-{b} (size {s})"` per
   `_call_llm`; outcome `llm-error` on exception, `dropped-items` when some lines
   were not returned/validated, else `ok`. The **summary-LLM**
   (`stage2/summary_llm.py`) is a single call → unit = `"summary_llm"`.

Because each stage function already injects `call_llm`, tests pass a fake
`call_llm` + a capturing recorder for deterministic assertions.

### CLI wiring

Each LLM stage CLI (`stage2/cli.py`, `summary/cli.py`, `plan/cli.py`,
`director/cli.py`, `guided_edit/cli.py`):

1. reads `general.llm_report` / `general.llm_report_dir` from config,
2. adds a `--llm-report-dir` override flag,
3. constructs `Recorder(stage=<name>, report_dir=..., enabled=...)`,
4. calls `recorder.clear()` before work (refresh own section),
5. passes the recorder into the stage function(s),
6. calls `rebuild_index(report_dir)` after its work so `index.md` always
   reflects the latest run of every stage.

`run_pipeline.sh` needs no functional change (config default
`output/llm_report` works from project root); `--llm-report-dir` is forwarded
for clarity/consistency with `--log-file`.

### Config

Add to `DEFAULTS["general"]` and `config.example.yml`:

```yaml
general:
  llm_report: true              # write the per-call LLM report under output/llm_report/
  llm_report_dir: "output/llm_report"
```

## Rerun behavior

Each stage `clear()`s only its own `report_dir/<stage>/` subdir at start and
rewrites it, then rebuilds `index.md`. A `--from-stage director` rerun refreshes
`director/` (and later stages) while reused stages keep their previous detail
files; the rebuilt index reflects all of them. No stale duplicates.

## Error handling

The report is best-effort and must never break the pipeline: a failure to write
a report file is caught and logged as a warning, and the stage proceeds. When
disabled, `NULL_RECORDER` makes every call a no-op.

## Testing (TDD)

Following repo policy (write tests first; for already-existing code, verify each
test catches a mutated implementation before reverting):

**`tests/test_llm_report.py`** — `Recorder` unit tests:
- `clear()` wipes only its stage subdir and is idempotent.
- `attempt()` + `flush_unit()` produce front-matter with the right
  `attempts`/`outcome`/`reason` and a body section per attempt with full
  prompt/response.
- `rebuild_index()` builds rows from front-matter across multiple stages; a
  re-run that replaces one stage's files updates only those rows.
- Disabled recorder writes nothing.
- Slugification handles units like `"lines 11-20 (size 10)"`.

**Per-stage tests** (extending each stage's existing test dir) assert the
recorder receives the correct `(attempts, outcome, reason)` via a fake
`call_llm` + capturing recorder, for: clean success, success-after-retry,
all-retries-fail, valid-empty (`ok-empty`), and dropped-items.

Each new assertion is verified to fail against a deliberately mutated
implementation (e.g. record wrong outcome, skip a flush, off-by-one attempt
count), then reverted.
